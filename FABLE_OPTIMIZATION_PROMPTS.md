# Brindlewick — Fable 5 Optimization & World-Building Prompt Series

How to use this: Fable 5 has finite usage per session. Each phase below is a **separate, self-contained prompt** sized to finish inside one session, including its own build/lint checks and a written handoff. Paste phases one at a time, in order, into fresh Fable 5 sessions (or the same session if it's still under budget — but don't force a phase to finish if it's running long; the handoff mechanism exists so you can stop and resume cleanly).

Each phase ends with Fable writing `docs/fable/PHASE_N_HANDOFF.md` — a short record of what changed, what it decided, what's left, and any manual step you need to take (e.g. running a migration). Phase N+1 starts by reading that file, so continuity survives the session boundary.

Do not skip Phase 1. Everything after it depends on the checklist it produces.

```
You are reviewing Brindlewick, a cozy mystery text adventure (Next.js 16 App Router + Supabase + TypeScript), with NPC dialogue powered by Claude Haiku via @anthropic-ai/sdk. - src/lib/anthropic/client.ts
- REVIEW_PROMPT.md (a prior review prompt for this repo — treat its architecture overview as background, but do not just re-run its Part 1/Part 2 analysis; this engagement goes further, into implementation)

Also skim: content/citizens/*.json, content/locations.json, content/items.json, supabase/migrations/*.sql (to understand current schema for citizens, locations, items, and how migrations are structured/numbered in this repo).

Then produce `docs/fable/MASTER_PLAN.md` covering four workstreams. For each item below, give: what it is, where it lives (file + function/line range), and a rough size estimate (S/M/L) for implementing it in a later phase.

**Workstream A — Performance & Supabase query efficiency**
Find sequential-await chains that should be Promise.all, repeated fetches of the same data within one request, N+1 patterns, and unnecessary round-trips (e.g. the client firing GET /api/game/state after every POST /api/game/command). Note the `allLocations` query in `handleConversationMessage` (engine.ts) that fetches all locations on every chat message as a specific example to investigate.

**Workstream B — Claude API call reduction**
Inventory every code path that calls `client.messages.create`: what triggers it, how often per session, and roughly how many tokens the system prompt carries (dialogue.ts's `continueConversation` sends the full 35-person roster and full location directory with hours on every turn — quantify this in tokens). Include `detectAndStorePlayerFact` in gossip.ts, which calls Haiku on every conversational message that matches a keyword filter. For each call site, note whether the call could be skipped, cached, batched, or replaced with scripted content, without a plan for HOW yet (that's later phases) — just flag it and estimate impact (calls/session saved).

**Workstream C — Dynamic world-entity system (the priority workstream)**
This is the main problem to solve: NPC dialogue is free-text generated by Claude, so NPCs sometimes reference places, people, or objects that don't exist in the game's database (locations, citizens, items tables). Right now `dialogue.ts` tells the model "never invent people" / "never invent location IDs" for the roster and location directory, but there is no equivalent constraint for items, and there is no mechanism at all for capturing something an NPC does invent so it becomes real and consistent.

Study the existing `[ESCORT:location_id]` and `[SUMMON:citizen_id]` tag mechanism in `dialogue.ts` (system prompt instructions) and `engine.ts` (`handleConversationMessage`, regex parsing of the tags, e.g. lines ~644-670 and ~817-849 as of this writing — confirm current line numbers). This is the pattern to extend, not replace: it already proves that Claude can reliably emit a structured tag inline in an otherwise-freeform response, at zero extra API cost, and the engine already knows how to parse and strip such tags.

Design (in the plan doc, not yet in code) a `[NEW_PERSON:...]` / `[NEW_PLACE:...]` / `[NEW_ITEM:...]` tag family that lets an NPC introduce a new entity into the story and have the game capture it into the database in the same turn, using only the content already in that one Claude response (no second API call). Cover in the plan:
- Exact tag payload format (e.g. `[NEW_PERSON:Aunt Clara|Marigold's aunt, runs the flower stand on weekends]`) and how it's parsed/stripped in engine.ts alongside the existing ESCORT/SUMMON parsing.
- What table(s) capture this. Options to weigh: inserting directly into `citizens`/`locations`/`items` with a new `source` column (e.g. `'seeded' | 'ai_generated'`) so they're immediately queryable by the existing `findCitizenByName` / `findLocationByName` / `findItemByName` / `getTownRoster` helpers and show up for every other NPC going forward — vs. a separate staging table that requires a promotion step. Recommend one, with reasoning tied to the user's requirement that the new entity be "referenced by the player and all NPCs later" without manual intervention.
- Sensible defaults for the columns a real row needs that the tag payload won't provide (a new location needs exits or it's unreachable — should it auto-attach an exit from the citizen's current location? should it start `is_hidden: true` until connected? a new citizen needs `tier`, `trust_max`, `home_location` — what are safe defaults? a new item needs a `location_id` — should it default to wherever the conversation is happening?).
- Deduplication against existing named entities (don't let "Marigold" get re-created because of a nickname mismatch) — reuse or extend the existing name-matching logic in `findCitizenByName`/`findLocationByName`/`findItemByName`.
- Guardrails: a per-session or per-day cap on auto-created entities, a check against TONE_CONTEXT rules in dialogue.ts (no danger/hostility), and a way for you (Rich) to review/curate AI-generated entities later — flag this for the admin dashboard (`src/components/admin/AdminDashboard.tsx`, `src/app/api/admin/*`).
- How the *other* half of the fix works: reducing how often NPCs invent things in the first place by improving what they're grounded in (this feeds Workstream D below — don't over-design it here, just note the connection).

**Workstream D — World-class feature gaps**
Skim REVIEW_PROMPT.md's Part 2 categories (NPC depth/memory, world liveness/time, player agency/discovery, narrative/atmosphere, polish/QoL) as a menu. You don't need to re-derive the list — instead, rank the existing suggestions (or ones you think are missing) by impact-per-effort, and pick a top 8-10 that later phases should actually build, not just suggest. Bias toward features that reinforce Workstream C (a world that remembers and grows) over features that are cosmetic.

**Output requirements for docs/fable/MASTER_PLAN.md:**
- One checklist per workstream, ordered by priority within the workstream.
- Each checklist item sized S (< 1 hour of focused work), M (a session), or L (needs its own phase/session).
- An explicit "Phase 2, 3, 4..." grouping proposal — bundle checklist items into future session-sized phases (aim for phases that touch 3-6 files and 3-6 checklist items each; implementation work needs smaller batches than this read-only audit did).
- Flag anything that needs a decision from Rich before it can be implemented (rather than guessing) — list these as open questions at the top of the doc.

Do not write or modify any application code in this phase. When done, write docs/fable/PHASE_1_HANDOFF.md summarizing what you found and confirming MASTER_PLAN.md is ready for Phase 2.
```

## Phase 3A — World-entity system: schema + creation pipeline

```
Continue work on Brindlewick. Read docs/fable/MASTER_PLAN.md (Workstream C) and all prior docs/fable/PHASE_*_HANDOFF.md files first.

This phase builds the plumbing only — no dialogue/prompt wiring yet (that's Phase 3B). Goal: a working, testable function that takes a tag-style mention (person, place, or item — name + short description + who mentioned it + context) and creates a real, immediately-queryable row for it.

1. Write the migration (follow the existing numbering convention in supabase/migrations/) adding whatever schema Workstream C settled on — most likely a `source` column (`'seeded' | 'ai_generated'`) on `citizens`, `locations`, and `items`, plus columns to track provenance (which citizen introduced this entity, in what conversation/context, when). Include sensible defaults per the plan (auto-generated locations start `is_hidden: true` until connected by an exit; auto-generated items default `location_id` to wherever they were mentioned; auto-generated citizens get safe `tier`/`trust_max`/`home_location` defaults).

2. In world.ts (or a new src/lib/game/world_entities.ts if that keeps things cleaner), write the creation functions: `createEntityFromMention(type, name, description, context)` or split per-type as the plan specifies. Must:
   - Dedupe against existing names using the same matching approach as `findCitizenByName`/`findLocationByName`/`findItemByName` (don't create "Aunt Clara" twice, don't create a citizen that collides with an existing nickname).
   - Enforce the guardrails from the plan (rate limit, tone check against TONE_CONTEXT rules already defined in dialogue.ts).
   - For a new location: auto-create a one-way or two-way exit from the current location (your call, document the reasoning) so it's actually reachable.
   - Return enough info (id, name) that a caller can reference the new entity immediately in the same response (e.g. "you can now go there").

3. Write a small standalone test script (scripts/test-entity-creation.ts, following the pattern of existing scripts/ files) that exercises creation for all three entity types plus the dedup path, and run it to confirm it works end-to-end against the dev Supabase instance.

Do not wire this into dialogue.ts or engine.ts's conversation handling yet — that's Phase 3B, so this phase can be verified in isolation first.

When done, write docs/fable/PHASE_3A_HANDOFF.md documenting the schema, the function signatures, the dedup approach, and the test script results. Flag any open question about default values you had to guess at, so Rich can review before Phase 3B wires it live.

## Phase 3B — World-entity system: live wiring into dialogue

```
Continue work on Brindlewick. Read all prior docs/fable/PHASE_*_HANDOFF.md files first, especially PHASE_3A_HANDOFF.md — the creation pipeline it describes is what this phase wires up live.

1. In dialogue.ts, extend the system prompts in both `generateNpcDialogue` and `continueConversation` (alongside the existing ESCORT/SUMMON tag instructions) with a `[NEW_PERSON:name|description]` / `[NEW_PLACE:name|description]` / `[NEW_ITEM:name|description]` tag family. Instruct the model: it's fine to mention a specific new person, place, or object if the story naturally calls for it (this is a good thing — it makes the world feel alive), but it MUST tag any such mention on first use so the town remembers it, using the same inline-tag convention as ESCORT/SUMMON. Keep the existing "never invent" language for the roster/location list, but reframe it as "don't invent people/places without tagging them" rather than "don't invent people/places at all."

2. In engine.ts's `handleConversationMessage` (and the `handleTalk`/`handleAsk` single-shot paths in dialogue.ts's `generateNpcDialogue`), parse and strip the new tags the same way ESCORT/SUMMON are parsed and stripped, and call the Phase 3A creation functions with the tag payload. On successful creation:
   - For a new item: if it's plausible for the NPC to be holding/offering it, consider integrating with the existing `npc_items.ts` holdings system rather than just floating it at the location — use judgment, document the choice.
   - For a new place: nothing further needed beyond the exit Phase 3A created — the player can now `go` there.
   - For a new person: nothing further needed — they now exist in the roster/`getTownRoster` and can be `find`-ed, `talk`-ed to (once at a location), or `summon`-ed like any other citizen.
   - Log the creation as a journal entry so the player sees it land narratively (e.g. "You've learned of a new place: ...") rather than silently.

3. Handle failure gracefully — if creation fails (guardrail rejection, dedup collision, DB error), the tag should be stripped and the conversation should continue without the game breaking; log it, don't surface an error to the player.

4. Playtest manually: have a conversation that provokes an NPC into mentioning something specific and new (ask an open-ended personal question), confirm the tag fires, confirm the entity is created, then start a **new** conversation with a **different** NPC and confirm they can be made aware of / reference the newly created entity (this is the actual bar for success — not just creation, but later referenceability).

Run `npm run build` and `npm run lint`. When done, write docs/fable/PHASE_3B_HANDOFF.md with the playtest transcript (both conversations) proving the loop works end to end, plus any tuning you did to the tag instructions to get reliable tagging behavior out of Haiku.
```

## Phase 4 — Context-efficient grounding (token & call reduction, now that the registry exists)

```
Continue work on Brindlewick. Read all prior docs/fable/PHASE_*_HANDOFF.md files first. Workstream C is done — there is now a live, growing registry of real entities (seeded + ai_generated) that NPCs can be grounded in. This phase uses that to cut the token cost identified in Phase 1's Workstream B, specifically the full 35-person roster and full 69-location directory sent on every `continueConversation` turn in dialogue.ts.

Design and implement a relevance-scoped context strategy instead of "send everything every turn": e.g., always include citizens/locations at or near the current location, citizens/locations mentioned earlier in this conversation's history, plus a compact index (name + one-line tag only, not full personality/hours) for everything else, with the full detail only pulled in if the topic seems to reference it (simple keyword match against the player's message is fine — no extra API call needed). Preserve the "don't invent" grounding property Workstream C relies on: the compact index must still cover the *entire* roster/location list by name, just without full detail for entries that aren't relevant this turn, so the tagging behavior from Phase 3B still has full coverage to dedupe against.

Also revisit `detectAndStorePlayerFact` in gossip.ts per whatever Phase 1 concluded (tighter keyword filter, dedup before the Haiku call rather than after, or folding fact-detection into the main dialogue call's response via a tag instead of a separate API call — consider whether a `[FACT:...]` tag on the main continueConversation response could replace this entirely, eliminating a whole class of Haiku calls the same way Workstream C eliminated the need for separate entity-extraction calls).

Measure and record actual token counts before/after (rough estimate is fine — count words/4 or use the Anthropic SDK's usage field on a real response) for a typical continueConversation call.

Run `npm run build` and `npm run lint`. When done, write docs/fable/PHASE_4_HANDOFF.md with before/after token estimates and confirmation that NPC grounding quality didn't regress (spot-check a few conversations against citizens/locations that are NOT near the player).

## Notes for Rich

- **Why the tag-based approach for Workstream C:** the codebase already proves this pattern works (ESCORT/SUMMON) and it's the only approach that adds *zero* extra Claude API calls — the entity gets created from content the model already generated for the player-facing response, rather than requiring a second "extraction" call after the fact. That directly serves the "minimize API calls" goal while solving the hallucination problem.
- **Why Phase 3 is split into three sub-phases:** it's the highest-risk, highest-value piece (schema + live-wiring + safety), and each sub-phase is independently verifiable before the next depends on it. If Fable 5 runs short on a sub-phase, you lose less work than if it were one large phase.
- **Decision points flagged for you across the phases:** whether auto-created locations start hidden until connected, how aggressively to rate-limit entity creation, and whether ai_generated entities need your approval before becoming fully live or can go live immediately with after-the-fact review. Phase 1's plan should surface these explicitly — read it before Phase 3A starts if you want to weigh in rather than let Fable decide.
- **If a phase runs out of usage mid-way:** it's fine — the handoff-doc pattern means you can resume in a new session by pasting the next phase's prompt as-is (it's written to read prior handoffs first). If a phase stops with no handoff doc at all, just tell the next session "Phase N didn't finish — check git status/diff, write the handoff for what's there, then continue."
